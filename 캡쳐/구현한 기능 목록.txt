구현한 기능 목록
이번 프로젝트에서는 Firebase Realtime Database를 이용하여 유저 데이터 저장, 아이템 구매, 인벤토리 관리, 게임 결과 저장, 거래소/경매장 기능을 구현하였다.
먼저 회원가입 기능에서는 사용자가 닉네임을 입력하면 Firebase의 UserInfo에 새로운 유저 데이터를 생성하도록 하였다. 유저 데이터에는 닉네임, 코인, 최고 점수, 보유 유닛 목록, 인벤토리 정보가 저장된다.
로그인 기능에서는 Firebase에서 입력한 닉네임과 일치하는 유저를 검색한 뒤, 해당 유저의 고유 키를 PlayerPrefs에 저장하였다. 이후 상점, 인벤토리, 결과 저장, 경매장 기능에서는 이 UserKey를 이용해 현재 로그인한 유저의 데이터를 불러온다.
상점 기능에서는 사용자가 코인을 사용해 아이템이나 유닛을 구매할 수 있도록 구현하였다. 아이템을 구매하면 Firebase에 저장된 유저의 코인이 감소하고, 인벤토리 JSON 데이터 안의 아이템 개수가 증가한다. 유닛을 구매하면 유저의 코인이 감소하고, UnitList에 해당 유닛의 보유 상태가 true로 변경된다.
인벤토리 기능에서는 Firebase에 저장된 Inventory JSON 문자열을 불러와 Dictionary 형태로 변환한 뒤, 각 아이템의 개수를 UI에 표시하도록 구현하였다. 아이템 사용 시 해당 아이템 개수를 감소시키고, 다시 JSON 문자열로 변환하여 Firebase에 저장한다.
게임 결과 저장 기능에서는 게임이 끝났을 때 보상 코인을 지급하고 점수를 Firebase에 저장하도록 구현하였다. 현재 점수는 CurrentScore에 저장하고, 기존 최고 점수보다 높은 경우에만 BestScore를 갱신하도록 하였다. 테스트를 위해 버튼을 누를 때마다 랜덤 점수가 생성되도록 하였으며, 보상 코인은 고정값으로 지급되도록 하였다.
추가로 거래소/경매장 기능을 구현하였다. 유저는 자신의 인벤토리에 있는 아이템을 판매 등록할 수 있고, 다른 유저는 등록된 아이템을 구매할 수 있다. 판매 등록 시 판매자의 인벤토리에서 해당 아이템 개수가 감소하고, Firebase의 AuctionItems에 판매글이 생성된다. 구매가 이루어지면 구매자의 코인이 감소하고 구매자 인벤토리에 아이템이 추가되며, 판매자의 코인이 증가하고 판매글 상태가 Sold로 변경된다.



2. 본인이 만든 아이템 3개 이상 설명
1) Antidote
Antidote는 해독제 아이템이다. 게임 내에서 오염 수치나 중독 상태를 회복하는 용도로 사용할 수 있는 아이템으로 설정하였다. 상점에서 코인을 사용해 구매할 수 있으며, 인벤토리에 저장된 개수가 1개 이상일 때 사용할 수 있다. 사용하면 인벤토리의 Antidote 개수가 1개 감소한다.
2) Degassing
Degassing은 가스 제거 아이템이다. 게임 내에서 가스나 오염 요소를 제거하는 도구로 사용될 수 있도록 구성하였다. 상점에서 구매 가능하며, 구매 시 Firebase에 저장된 Inventory 데이터에서 Degassing 개수가 증가한다. 사용 시 개수가 감소하고 변경된 인벤토리 정보가 Firebase에 다시 저장된다.
3) Spanner
Spanner는 수리용 아이템이다. 파이프나 고장난 오브젝트를 고치는 기능과 연결할 수 있는 아이템이다. 가격은 다른 아이템보다 낮게 설정하여 초반에도 쉽게 구매할 수 있도록 하였다. 인벤토리에 저장된 Spanner 수량이 0보다 클 때 사용할 수 있다.
4) ShotGun
ShotGun은 공격용 아이템이다. 다른 아이템보다 가격이 높게 설정되어 있으며, 강력한 장비 아이템으로 사용할 수 있다. 상점에서 구매하면 인벤토리의 ShotGun 개수가 증가하고, 사용하면 개수가 감소한다.

4. Firebase 데이터 구조 설명
Firebase Realtime Database의 기본 유저 데이터는 UserInfo 아래에 저장된다. 각 유저는 Firebase에서 생성된 고유 키를 가지고 있으며, 그 안에 닉네임, 코인, 점수, 유닛 목록, 인벤토리 정보가 저장된다.
예시 구조는 다음과 같다.
{
  "UserInfo": {
    "UserKey": {
      "NickName": "PlayerName",
      "Coin": 1000,
      "Score": 0,
      "BestScore": 0,
      "CurrentScore": 0,
      "UnitList": "{\"Unit1\":true,\"Unit2\":false,\"Unit3\":false,\"Unit4\":false,\"Unit5\":false,\"Unit6\":false}",
      "Inventory": "{\"Antidote\":0,\"Degassing\":0,\"Spanner\":0,\"ShotGun\":0}"
    }
  }
}
NickName은 유저의 닉네임이다.
Coin은 유저가 보유한 코인이다.
Score는 기존 점수 필드이며, 이후 BestScore와 같은 최고 점수 용도로 사용할 수 있다.
BestScore는 지금까지 기록한 최고 점수이다.
CurrentScore는 가장 최근 게임에서 얻은 점수이다.
UnitList는 유저가 보유한 유닛 목록을 JSON 문자열로 저장한 값이다.
Inventory는 유저가 보유한 아이템 목록을 JSON 문자열로 저장한 값이다.
경매장 기능을 위해서는 AuctionItems라는 별도의 데이터 구조를 추가하였다.
{
  "AuctionItems": {
    "AuctionKey": {
      "ItemName": "Antidote",
      "Count": 1,
      "Price": 500,
      "SellerKey": "판매자 유저 키",
      "SellerNickName": "판매자 닉네임",
      "BuyerKey": "구매자 유저 키",
      "BuyerNickName": "구매자 닉네임",
      "Status": "Sold",
      "CreatedAt": "2026-06-17 12:00:00",
      "SoldAt": "2026-06-17 12:10:00"
    }
  }
}
ItemName은 판매 등록된 아이템 이름이다.
Count는 판매 수량이다.
Price는 판매 가격이다.
SellerKey는 판매자의 Firebase 유저 키이다.
SellerNickName은 판매자의 닉네임이다.
BuyerKey는 구매자의 Firebase 유저 키이다.
BuyerNickName은 구매자의 닉네임이다.
Status는 판매 상태이며, 판매 중이면 Selling, 판매 완료면 Sold로 저장된다.
CreatedAt은 판매 등록 시간이고, SoldAt은 판매 완료 시간이다.


5. PlayerPrefs에 저장한 값 설명
로그인 또는 회원가입이 성공하면 현재 유저 정보를 로컬에 저장하기 위해 PlayerPrefs를 사용하였다.
저장한 값은 다음과 같다.
Key	저장 값	사용 목적
UserKey	Firebase에서 생성된 유저 고유 키	현재 로그인한 유저의 Firebase 데이터를 찾기 위해 사용
UserNickName	유저 닉네임	UI 표시 및 경매장 판매자/구매자 이름 저장에 사용
UserKey는 상점, 인벤토리, 결과 저장, 경매장 기능에서 가장 중요한 값이다. Firebase의 UserInfo/UserKey 경로를 찾을 때 사용되기 때문에, 로그인 정보가 없으면 데이터 저장이나 불러오기를 진행할 수 없다.
UserNickName은 경매장 기능에서 판매자 닉네임과 구매자 닉네임을 표시하기 위해 사용하였다.
6. Inventory JSON 처리 방식 설명
Unity의 기본 JsonUtility는 Dictionary 자료형을 직접 저장하고 불러오는 데 불편함이 있다. 그래서 인벤토리와 유닛 목록은 Newtonsoft.Json을 사용하여 Dictionary를 JSON 문자열로 변환한 뒤 Firebase에 저장하였다.
인벤토리는 코드 내부에서 다음과 같은 Dictionary 형태로 사용된다.
Dictionary<string, int> inventory = new Dictionary<string, int>();
예를 들어 기본 인벤토리는 다음과 같다.
inventory["Antidote"] = 0;
inventory["Degassing"] = 0;
inventory["Spanner"] = 0;
inventory["ShotGun"] = 0;
Firebase에 저장할 때는 Dictionary를 JSON 문자열로 변환한다.
string inventoryJson = JsonConvert.SerializeObject(inventory);
Firebase에는 다음과 같은 문자열 형태로 저장된다.
"{\"Antidote\":0,\"Degassing\":0,\"Spanner\":0,\"ShotGun\":0}"
반대로 Firebase에서 불러올 때는 JSON 문자열을 다시 Dictionary로 변환한다.
inventory = JsonConvert.DeserializeObject<Dictionary<string, int>>(inventoryJson);
이 방식을 사용하면 Firebase에는 문자열로 저장되지만, Unity 코드에서는 Dictionary처럼 쉽게 아이템 개수를 증가시키거나 감소시킬 수 있다.
예를 들어 아이템을 구매하면 다음과 같이 개수를 증가시킨다.
inventory[itemName]++;
아이템을 사용하거나 판매 등록하면 다음과 같이 개수를 감소시킨다.
inventory[itemName]--;
변경된 Dictionary는 다시 JSON 문자열로 변환하여 Firebase에 저장한다.

7. 실행 중 발생한 문제와 해결 방법
1) 결과 저장 시 “게임 결과 저장 중...”에서 멈춘 문제
처음에는 게임 결과를 저장할 때 화면에 “게임 결과 저장 중...”까지만 출력되고 완료 메시지가 나오지 않는 문제가 있었다. 원인은 저장 실패 또는 데이터 불러오기 실패가 발생했을 때 실패 메시지를 UI에 표시하지 않았기 때문이다.
해결 방법으로 IsFaulted, IsCanceled 조건에서 SetMessage()를 호출하여 오류 메시지를 출력하도록 수정하였다. 또한 Debug.LogError()를 사용해 Firebase 작업 중 발생한 예외를 확인할 수 있도록 하였다.
________________________________________
2) Firebase Database URL이 서로 달라 데이터가 없는 문제
로그인, 상점, 인벤토리 스크립트와 결과 저장 스크립트에서 사용하는 Firebase Database URL이 서로 달라서 로그인한 유저의 UserKey로 데이터를 찾지 못하는 문제가 발생하였다.
해결 방법으로 모든 Firebase 관련 스크립트의 databaseUrl을 같은 주소로 통일하였다. 같은 Firebase 프로젝트의 같은 Realtime Database를 사용해야 UserKey로 동일한 유저 데이터를 찾을 수 있다.
________________________________________
3) UnityMainThreadDispatcher 오브젝트가 없어 발생한 오류
씬을 이동했다가 다시 게임 씬으로 돌아왔을 때 다음과 같은 오류가 발생하였다.
UnityMainThreadDispatcher could not find the UnityMainThreadDispatcher object.
Please ensure you have added the MainThreadExecutor Prefab to your scene.
Firebase의 비동기 작업이 끝난 뒤 Unity UI를 수정하려면 메인 스레드에서 실행되어야 한다. 이를 위해 UnityMainThreadDispatcher를 사용했는데, 해당 씬에 MainThreadExecutor 프리팹이 없어서 오류가 발생하였다.
해결 방법으로 Firebase를 사용하는 씬마다 MainThreadExecutor 프리팹을 추가하였다. 또한 Awake()에서 바로 UnityMainThreadDispatcher.Instance()를 호출하면 실행 순서 문제가 생길 수 있으므로, 필요에 따라 Start()에서 초기화하도록 수정할 수 있다.
________________________________________
4) 점수가 계속 300으로 보이는 문제
처음에는 결과 저장 시 계속 300점만 나오는 것처럼 보였다. 하지만 실제로 300은 점수가 아니라 보상 코인 값이었다.
[SerializeField] int clearRewardCoin = 300;
해결 방법으로 저장 완료 메시지에 이번 점수, 획득 코인, 현재 코인, 최고 점수를 모두 따로 표시하도록 수정하였다. 또한 테스트를 위해 버튼을 누를 때마다 랜덤 점수가 생성되도록 Random.Range()를 사용하였다.
________________________________________
5) 최고 점수와 현재 점수를 따로 저장하는 문제
처음에는 점수 필드가 Score 하나만 있어서 이번 게임 점수와 최고 점수를 구분하기 어려웠다.
해결 방법으로 Firebase에 CurrentScore와 BestScore 필드를 추가하였다. CurrentScore는 매 게임 결과마다 갱신하고, BestScore는 기존 최고 점수보다 현재 점수가 높을 때만 갱신하도록 하였다.
________________________________________
6) 경매장 ScrollView에서 아이템이 아래로 정렬되지 않는 문제
경매장에 판매글이 여러 개 등록되었을 때 ScrollView의 Content 안에서 아이템들이 아래로 내려가지 않거나 겹쳐 보이는 문제가 있었다.
원인은 ScrollView의 Content 오브젝트 높이가 아이템 개수에 맞게 자동으로 늘어나지 않았기 때문이다.
해결 방법으로 Content 오브젝트에 Vertical Layout Group과 Content Size Fitter를 추가하였다. Content Size Fitter의 Vertical Fit을 Preferred Size로 설정하고, 판매글 프리팹에는 Layout Element를 추가하여 Preferred Height를 지정하였다. 이를 통해 판매글이 여러 개 생성되어도 자동으로 아래로 정렬되고 스크롤이 가능해졌다.
________________________________________
7) 경매장에서 판매자와 구매자의 데이터가 동시에 바뀌는 문제
경매장 구매 기능은 구매자 코인 감소, 구매자 인벤토리 증가, 판매자 코인 증가, 판매글 상태 변경이 한 번에 이루어져야 한다. 처음에는 각각 따로 저장하면 중간에 오류가 발생했을 때 데이터가 꼬일 수 있는 문제가 있었다.
해결 방법으로 Firebase의 UpdateChildrenAsync()를 사용하여 여러 경로의 데이터를 한 번에 갱신하도록 구현하였다. 예를 들어 구매가 완료되면 구매자의 Coin, 구매자의 Inventory, 판매자의 Coin, 판매글의 Status, BuyerKey, BuyerNickName, SoldAt을 한 번에 업데이트한다.
이를 통해 거래소 기능에서 구매자와 판매자의 데이터가 함께 갱신되도록 만들었다.
